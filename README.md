英会話スクールの生徒専用学習管理アプリ


## 📋 プロジェクト概要

**KORU 学習支援・運用統合アプリ（Web）**

Pythonを用いたバックエンドとAI機能を中核に、英語学習体験と運用業務を一体化する包括的な学習支援プロダクト。

### 主な機能

### ① 予約管理

- レッスン予約管理（**Google Meet / Zoom 自動連携**）
- 予約完了時にミーティングリンクを自動生成・通知
- 生徒ダッシュボードに「次のレッスン」を常時表示

### ② Conversation Class 管理

- Conversation Class の予約管理（**Zoom 自動連携**）
- **チューター向け管理画面**
    - 対応可能な日時の登録
    - 生徒側予約画面へ自動反映
- 生徒・チューター双方のスケジュール可視化

### ③ 教材管理

- Notion連携による教材閲覧・管理
- レッスン後のまとめノート共有

### ④ 単語帳

- 内製単語帳機能
- Quizlet連携（将来的に検討）
- レッスン中に「言えなかった単語」を自動保存

### ⑤ 学習コミュニケーション

- Webアプリ内メッセージ機能
- 公式LINEからの脱却
- 学習連絡・簡易フィードバックを一元化

### ⑥ レポート・カリキュラム管理

- レッスンレポート閲覧
- クールごとの学習カリキュラム管理

### ⑦ 学習目標・達成度の可視化

- 目標設定と進捗表示
- 日次／週次の学習タスク完了状況を可視化

### ⑧ AI学習支援機能

- **英作文のAI添削**
- **作文・会話ログからの語彙自動抽出**
- **AI音声生成による発音確認**
- 学習履歴と連動した復習体験の提供

## 🎯 目的・ゴール

### ターゲット

- KORUの生徒（継続的な学習が必要な英語学習者）

### 課題

- 予約管理・変更対応の運用負荷が高い（レッスン／TenTalk）
- 学習が点在し、進捗や成果が把握しづらい
- 公式LINE中心の運用によるUX低下（動作の重さ／ファイル不安定）

### ゴール

- 学習・予約・コミュニケーションを一つの体験に統合
- AIを活用し、復習・定着までを自動化・効率化

### 成功指標（KPI）

- 予約変更に要する工数の削減率
- 週次アクティブ率（WAU）
- 学習タスク完了率／継続率

## 🏛️ アーキテクチャ

- **SPA + API 構成**
- Frontend（Vercel） → Backend API（Render／Railway） → PostgreSQL
- 外部連携：Google Meet／Zoom／Notion／Quizlet（任意）

```
[React / TypeScript]
        │ HTTP / JSON
        ▼
[Django REST API]
        │
        ├─ PostgreSQL
        ├─ AI Service（添削・語彙抽出・音声生成）
        ├─ Zoom / Google Meet API
        ├─ Notion API
        └─ Quizlet API（検討）

```

### 設計方針

- 機能単位で疎結合（予約／学習／AI／コミュニケーション）
- **AI処理はサービス層として分離**
- まずはMVP：予約・学習進捗・メッセージ機能
- 外部APIは Adapter 層で抽象化し変更に強い構成

## UI / UX

### 全体方針

- **日常的に使いたくなる分かりやすさ**を最優先。ポートフォリオ用途に寄せすぎず、実運用を前提にしたUI。
- ヤオコー公式アプリのような、**情報量は多いが迷わない設計**を参考にする。
- 学習・予約・確認といった主要行動が、**最短2タップ以内**で完結する導線を目指す。

### ダッシュボード（ホーム）

- ファーストビューに「**次のレッスン**」を大きく表示。
    - 日時／講師名／形式（オンライン・対面）を一目で把握できる。
- クレジットカードアプリの**カードUI風デザイン**を採用。
    - 視覚的に“今一番大事な情報”だと直感できる。
    - 横スワイプで「次」「その次」の予約も確認できる設計も検討。

### 共有・コミュニケーション導線

- 「次のレッスン」カード内に**共有ボタン**を配置。
    - ワンタップで「メールに送信」できる導線。
    - 将来的にLINE・カレンダー連携も拡張可能な設計を想定。
- 共有時は、受信者側で内容がすぐ理解できる**シンプルなレイアウト**を意識。

### 予約管理UI

- 予約一覧は「次回予約」「過去のレッスン」を明確に分離。
- 編集・キャンセルは誤操作を防ぐため、**カード内に直接置かず、詳細画面に集約**。

### Figmaデザイン方針

- 先に**ダッシュボードUIを完成**させ、そこから予約管理へ展開。
- コンポーネント化（カード／ボタン／ステータス表示）を前提にデザイン。
- 実装（React）を見据えた、余白・フォントサイズ・状態（hover / disabled）を定義。

---

※ UIは「作っていて楽しい」「使っていて気持ちいい」を重要指標とし、実装フェーズでUX改善を前提に継続調整する。
